## 3. 关键教训

### 教训 22: 代码审查必须分维度

以前的审查是"看一遍代码找问题"，结果遗漏系统性缺陷。本轮分 3 个维度审查：

1. **C 语言安全** — 整数溢出、指针别名、未初始化变量
2. **嵌入式专项** — 中断优先级、DMA 配置、时钟树
3. **并发安全** — 竞态条件、内存屏障、volatile

每个维度独立审查，交叉验证。**分维度审查比"看一遍"多发现 3-5 倍问题。**

### 教训 23: ISR 和任务之间的数据传递必须有内存屏障

```c
// ❌ 错误：可能乱序读取
uint16_t *buf = process_ptr;   // 可能读到旧值
uint16_t len = process_len;    // 可能读到新值

// ✅ 正确：反序读取 + 内存屏障
uint16_t len = process_len;
__DMB();
uint16_t *buf = process_ptr;
__DMB();
```

ARM Cortex-M 是弱序架构。ISR 写入 `process_ptr` 和 `process_len` 的顺序，任务不一定按相同顺序读取。`__DMB()` 强制顺序。

### 教训 24: Stop 函数必须与 Start 函数对称

`HAL_TIM_Base_Start()` 用轮询模式启动，`HAL_TIM_Base_Stop_IT()` 用中断模式停止 — 不匹配。

**规则：Start 用什么函数，Stop 就用对应的函数。**

| Start | 对应 Stop |
|-------|-----------|
| `HAL_TIM_Base_Start()` | `HAL_TIM_Base_Stop()` |
| `HAL_TIM_Base_Start_IT()` | `HAL_TIM_Base_Stop_IT()` |
| `HAL_TIM_Base_Start_DMA()` | `HAL_TIM_Base_Stop_DMA()` |

### 教训 25: 半传输中断不是真正的乒乓双缓冲

当前方案：单缓冲区 `adc_buffer[1024]` + 半传输/全传输中断。

```
DMA 写入: [===== 512 =====][===== 512 =====]
任务处理: [正在处理前半][正在处理后半]
```

问题：如果任务处理时间 > 半缓冲区填充时间，DMA 会覆写正在处理的数据。

当前安全（10kHz 采样率，51.2ms 窗口 >> 10ms 处理时间），但提高采样率后会出问题。

**未来改进**：真正的乒乓双缓冲（两个独立缓冲区，DMA 自动切换）。

### 教训 26: 任务栈使用要估算上限

`Osc_HandleAdcData()` 局部变量 + 嵌套调用，估计 1.5-2KB。任务栈 2KB 接近上限。

**估算方法**：
1. 最深调用链的所有局部变量大小之和
2. 加上中断嵌套的额外开销（~256 bytes）
3. 乘以 1.5 安全系数

当前改为 4KB，安全。
