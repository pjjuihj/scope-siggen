## 一、犯了什么错

### 1. 配置类 Bug 占 80%

| Bug | 耗时 | 根因 |
|-----|------|------|
| TIM8 TRGO 设为 RESET | 2h | 没查 RM0090 默认值 |
| ADC DMA 未配置 | 1.5h | CubeMX 生成后没验证 |
| DMAContinuousRequests=DISABLE | 1h | 不知道这个参数的作用 |
| ADC 采样时间太短 | 1h | 不懂 DAC 输出阻抗 |
| TIM5 预分频公式错误 | 2h | 忘记考虑 ARR |
| sinf() 返回垃圾值 | 1h | 不了解 ARM CC 浮点库 |
| OLED I2C 任务竞争 | 1.5h | 没考虑并发安全 |

**总耗时**: ~10 小时配置调试 vs ~2 小时功能开发

### 2. 不读手册就写代码

TIM5 预分频公式：
```
错误: PSC = timer_clk / target_trgo - 1
正确: PSC = timer_clk / (target_trgo * (ARR+1)) - 1
```

如果读了 RM0090 的定时器章节，5 分钟就能发现这个错误。实际花了 2 小时排查。

### 3. 写太多代码再测试

第 1 天写了 400 行代码，编译通过，烧录后什么都不工作。花 3 小时排查 6 个配置 bug。

如果每写 20 行就测试，每个 bug 只需 5 分钟。

### 4. 调试时陷入循环

波形修复时：
- 尝试 1: 改正弦波生成 → 还是锯齿波
- 尝试 2: 用查表法 → 还是锯齿波
- 尝试 3: 用三角波 → 还是锯齿波
- 尝试 4: 用方波 → 发现 DAC 正常！
- 最终: 问题不在波形生成，在 TIM5 频率

**浪费了 1 小时在错误的方向上。**

### 5. 不分析数据就下结论

串口输出显示：
```
W:len=512 min=1775 max=1813 v=1500mV
```

我只看了 `v=1500mV`，说"电压正常"。没有分析 `min=1775 max=1813`（范围只有 38），说明 512 个样本几乎是一条直线。

### 6. 不更新文档

修复完成后不更新调试教训文档，需要用户提醒。违反了 embedded-workflow 的 Step 10。

### 7. 过度信任 AI 生成的代码

初始 CubeMX 配置有 6 个 bug，都是 AI 生成的。我没有逐项验证，直接写应用代码。

## 二、正确的做法

### Day 1: 不写一行应用代码

```
CubeMX 生成 → 编译 → 烧录 → 串口输出 "Hello"
→ 配置 ADC 轮询 → 串口输出 ADC 值
→ 配置 DAC 单次输出 → 万用表量引脚
→ 配置 TIM 触发 → 读 TIM->CNT
```

**只做一件事：确认每个外设单独工作。**

### Day 2: 打通数据链路

```
TIM8 → ADC → DMA → 中断 → 串口打印
TIM5 → DAC → DMA → 万用表
DAC 输出 → 导线 → ADC 输入 → 串口打印
```

**只做一件事：确认数据从 A 流到 B。**

### Day 3: 集成显示

```
OLED 初始化 → 画一条线
ADC 数据 → 波形渲染 → OLED 显示
```

**第三天才接 UI。**

## 三、关键教训

### 教训 1: 配置 Bug 靠测试预防，不靠调试解决

AI 生成的 CubeMX 配置可能有错误。烧录后立即读寄存器验证：
- TIMx->CNT 在变？
- DHR12R1 非零？
- CR bit0=1？

### 教训 2: 定时器公式必须考虑 ARR

```
TRGO = timer_clk / (PSC+1) / (ARR+1)
```

反推 PSC 时，必须乘以 `(ARR+1)`。

### 教训 3: ADC 缓冲区范围是 DAC 频率的直接指标

- 范围 ≈ 2048: 频率正确
- 范围 < 100: 频率太低

### 教训 4: 方波测试是验证 DAC/ADC 链路的最快方法

方波只有两个电平，容易验证。正弦波有连续变化，容易误判。

### 教训 5: ARM CC 浮点库不可靠

优先用查表法，不要依赖 `sinf()`、`cosf()`。

### 教训 6: 调试时先分析数据，再下结论

不要只看"电压正常"，要看 min/max 范围、差分、分布。

### 教训 7: 陷入循环时停下来

尝试 3 次还不行 → 停下来 → 重新分析数据 → 换方向。
