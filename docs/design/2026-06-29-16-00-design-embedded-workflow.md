## 1. 概述

### 1.1 目标

创建一个通用嵌入式开发工作流，整合六个信息源，实现闭环验证。

### 1.2 核心原则

1. **一切代码服务硬件** — 代码不是独立存在的，是为了控制硬件
2. **先读手册，再写代码** — 没有手册就去下载，这是硬性要求
3. **每写 20 行就验证** — 不要写 400 行再测试
4. **库有 bug** — 写代码前必须了解库的已知问题
5. **先硬件后软件** — 读寄存器判断硬件状态

### 1.3 开发规范

1. **小步迭代 MVP** — 每次只做一小块（~20 行），做完就验证
2. **主动拆分模块** — 大功能拆成小模块，每个模块独立可测
3. **限制 AI 修改范围** — 指定 AI 只改哪些文件，不让 AI 瞎改
4. **死守安全底线** — 不破坏已有功能，不引入新 bug
5. **科学应对报错** — 先分析原因，再修复代码，不要盲目试错
6. **两次修复无效 → 回滚** — 第一次让 AI 修复，第二次让 AI 再试，还不行回滚代码

### 1.4 防骗机制

**AI 不能说"我验证了"：**
- AI 只能输出代码和预期值
- AI 必须输出验证代码让用户运行
- AI 必须输出检查清单

**用户必须确认：**
- 关键步骤必须用户确认
- 用户说"确认"才能继续
- 用户说"不对"必须回退

**工具必须验证：**
- 编译器验证（0 errors, 0 warnings）
- 静态分析验证（cppcheck）
- 串口验证（读寄存器值）
- 万用表/示波器验证（硬件测量）

## 2. 六个信息源

### 2.1 数据手册（RM0090）— 必须读取

- 寄存器定义
- 配置流程
- 时序要求
- Errata（芯片勘误）

### 2.2 HAL 库手册（UM1725）— 必须读取

- 函数签名
- 参数说明
- 已知问题

### 2.3 网络资源 — 可选读取

- GitHub Issues（社区问题）
- ST Community（官方论坛）
- 技术博客

### 2.4 项目文档 — 必须读取

- 设计文档（docs/design/）
- 实现计划（docs/plan/）
- 调试日志（docs/debug_logs/）
- 报告（docs/reports/）
- 全局规则（CLAUDE.md）

### 2.5 约束文档 — 必须读取

- 硬件约束（MCU、外设、引脚）
- 软件约束（库、驱动、RTOS）
- 流程约束（验证、测试、提交）
- 资源约束（Flash、RAM）

### 2.6 时间约束 — 必须读取

- 项目截止日期
- 里程碑日期
- 任务时间估计
- 时间缓冲

## 3. 工作流

### 3.1 完整流程

```
Step 0: 读取全局规则（CLAUDE.md）
├─ 读取项目规则
├─ 读取开发规范
├─ 读取验证流程
└─ 读取约束条件

Step 1: 判断任务类型
├─ 新增外设 → 完整流程
├─ 修复 Bug → 简化流程
├─ 优化代码 → 快速流程
├─ 重构代码 → 安全流程
├─ 测试任务 → 测试流程
├─ 文档任务 → 文档流程
└─ 部署任务 → 部署流程

Step 2: 读数据手册（必须）
├─ 读 RM0090 → 找到寄存器定义
└─ 找到配置流程和时序要求

Step 3: 读 HAL 库手册（必须）
├─ 读 UM1725 → 找到函数说明
└─ 找到已知问题

Step 4: 读项目文档（必须）
├─ 读设计文档 → 了解项目架构
├─ 读调试日志 → 了解已知问题
├─ 读检查清单 → 了解应该检查什么
└─ 读约束文档 → 了解约束条件

Step 5: 搜索网络资源（可选）
├─ 检查 perplexity-search 是否可用
├─ 检查 agent-reach 是否可用
└─ 搜索已知问题和已验证配置

Step 6: 整合信息
├─ 数据手册 + HAL 库 + 项目文档 + 约束文档
├─ 交叉验证
└─ 生成配置清单

Step 7: 生成验证代码（自动）
├─ 基于 RM0090 生成寄存器验证代码
├─ 基于 UM1725 生成函数验证代码
└─ 基于已知问题生成检查代码

Step 8: 用户运行验证
├─ 用户: 编译烧录
├─ 用户: 运行验证代码
├─ 用户: 输出结果
└─ AI: 自动分析结果

Step 9: 修复（如果需要）
├─ AI: 分析失败原因
├─ AI: 生成修复代码
├─ AI: 编译验证
├─ 用户: 重新验证
└─ 两次修复无效 → 回滚代码

Step 10: 继续下一步
├─ 验收：上一步全部通过
└─ 有未通过项 → 不能继续
```

### 3.2 验收标准

| 步骤 | 验收标准 | 失败处理 |
|------|---------|---------|
| Step 0 | 读取 CLAUDE.md 并理解规则 | 重新读取 |
| Step 1 | 明确任务类型和流程类型 | 重新判断 |
| Step 2 | 输出具体页码和内容 | 回到 Step 2 |
| Step 3 | 输出具体页码和内容 | 回到 Step 3 |
| Step 4 | 输出具体文档和内容 | 回到 Step 4 |
| Step 5 | 输出搜索结果（如果可用） | 跳过 |
| Step 6 | 输出配置清单 | 回到 Step 6 |
| Step 7 | 编译 0 errors | 回到 Step 7 |
| Step 8 | 寄存器值正确 | 回到 Step 9 |
| Step 9 | 重新验证通过 | 回到 Step 9（最多 2 次）|
| Step 10 | 上一步全部通过 | 不能继续 |

## 4. 灵活性

### 4.1 根据任务类型调整

| 任务类型 | 流程 | 说明 |
|---------|------|------|
| 新增外设 | 完整流程 | 读手册 → 配置 → 验证 → 集成 |
| 修复 Bug | 简化流程 | 分析 → 修复 → 验证 |
| 优化代码 | 快速流程 | 修改 → 编译 → 完成 |
| 重构代码 | 安全流程 | 理解 → 修改 → 验证 |
| 测试任务 | 测试流程 | 设计测试 → 编写测试 → 运行测试 → 分析结果 |
| 文档任务 | 文档流程 | 收集信息 → 编写文档 → 审查 → 发布 |
| 部署任务 | 部署流程 | 准备 → 烧录 → 验证 → 确认 |

### 4.2 根据硬件平台调整

- STM32：使用 RM0090 + HAL 库
- ESP32：使用 ESP-IDF 文档
- NXP：使用 MCUXpresso 文档
- 其他：使用通用 Cortex-M 文档

### 4.3 根据用户经验调整

- 新手：完整流程 + 详细解释
- 中级：标准流程 + 关键提示
- 专家：快速流程 + 只检查关键点

### 4.4 限制 AI 修改范围

- 指定 AI 只改哪些文件
- 不让 AI 瞎改其他文件
- 每次改动范围明确
- 两次修复无效 → 回滚代码

## 5. 自动化

### 5.1 自动化验证步骤

- AI 生成验证代码
- AI 编译验证代码
- AI 分析结果
- 用户只运行验证代码

### 5.2 自动生成验证代码

- 基于 RM0090 生成寄存器验证代码
- 基于 UM1725 生成函数验证代码
- 基于已知问题生成检查代码

### 5.3 自动分析结果

- AI 解析寄存器值
- AI 对比预期值
- AI 判断是否正确

### 5.4 自动修复

- AI 分析失败原因
- AI 生成修复代码
- AI 编译验证
- 两次修复无效 → 回滚代码

### 6.1 CLAUDE.md 文件

位置：项目根目录 `CLAUDE.md`

- AI 不需要每次重复说明规则
